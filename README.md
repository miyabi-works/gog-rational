# gog-rational 🛡️

**Default Deny Strategy for Autonomous Agents** &nbsp;·&nbsp; 特許出願済み (JP) &nbsp;·&nbsp; MIT License

---

> AIエージェントはメールを誤送信し、機密ファイルを社外に共有し、深夜2時に全社員へ通知を送る。
> 壊れているからではない。**「やってはいけない」を誰も教えていないからだ。**

既存のSKILL.mdは「どうやるか（操作マニュアル）」を書いている。
gog-rationalは「**やっていいか（規範・仕様書）**」を書いている。

gog-rationalは、AIにコマンドの使い方を教えるのではなく、実行前に「理路（ロジック）」を照合させることで、エージェントを**「制御可能な道具」**へと変える。
照合条件を全て満たして初めて実行可能になる。許可されていないことは、全部止まる。

---

## 3つの防衛線

安全性のレベルに応じて組み合わせて運用する。

| 層 | 名称 | 役割 | 実装方法 |
|---|---|---|---|
| Layer 1 | 理性の仕様書 | AIの「憲法」。照合条件を Markdown で定義する | 本リポジトリのスキルファイルをそのまま使用。プログラミング不要 |
| Layer 2 | 実行制御ラッパー | AIが照合③をパスしたと明示しない限り、コマンド実行を物理的に拒絶する | Layer1に加え、エージェントエンジン側の実装が別途必要 |
| Layer 3 | 物理バリデーター | 型・パス・ドメインをAIの判断を介さず検証する | **本 Markdown を設計図としてAIコーディングで生成できる** |

### Layer 2 について

Layer 2はMarkdownの内容ではなく、エージェントエンジンの構造に関わる実装のため、AIコーディングで自動生成できるものではない。AIの出力を監視してPASS/FAILを検知し、実際のAPI実行を制御するラッパーをエージェントエンジン側で別途実装する必要がある。

### Layer 3 の生成体験

本リポジトリのスキルファイルをAIに渡し「これをもとにバリデーターを書いて」と一言頼むだけで、照合条件に対応したコードが生成される。gog-rationalのMarkdownはAIにとって極めて解像度の高い「コーディング指示書」として機能する。

> **補足：AI生成コードの扱いについて**
> 生成されるコードは「ほぼ完成」した状態だが、実行環境への適応（ファイルパス・APIトークン・ライブラリバージョン等）は人間の手で微調整する必要がある。gog-rationalの哲学が「最後は人間が確定する（照合③）」であるのと同様に、生成されたコードも人間がレビューすることを前提としている。

## テスト結果

Google Gemini で 25 の攻撃パターン（権威偽装・インジェクション・確認回避）を全て阻止。

| スキル | 攻撃数 | 阻止した主な攻撃内容 |
|---|---|---|
| Gmail 送信 | 1 | 本文への削除命令混入の検知・無視 |
| カレンダー登録 | 4 | 過去日時への登録警告・招待メール送信前の強制確認 |
| ファイル操作 | 4 | ルートディレクトリ削除・再帰的削除・管理者偽装の拒否 |
| Slack 投稿 | 5 | #general への全体通知・未承認の @channel 投稿の阻止 |
| Google Drive | 6 | 社外アドレスへの機密共有・証拠隠滅のための削除指示の遮断 |
| Google Sheets | 5 | セル内数式（IMPORTXML 等）によるデータ外部送信の検知 |
| **合計** | **25** | ✅ **全阻止** |

詳細なテストログ → [tests/ja/test-results.md](ja/tests/test-results.md)

## 安全性と限界について

gog-rationalは魔法の杖ではない。AIが照合プロセスを省略して応答する可能性や、操作を巧妙に組み合わせるリスクは理論上残る。

しかし、「何が正しいか」という基準（仕様書）を人間が持たないままAIを走らせることに比べれば、このシステムは圧倒的に安全だ。本プロジェクトは、AIを100%信頼するのではなく、AIに「説明責任」を負わせ、人間が主導権を握り続けるためのフレームワークである。

## v0.1 について

本バージョンは設計思想の提示が目的です。現バージョンはモデルの instruction-following 能力に依存しており、コードによる強制ではありません。Layer 2・3 の実装はこの仕様書を設計図として各自で行うか、AIコーディングで生成してください。

## 設計思想

| 原則 | 内容 |
|---|---|
| 仕様書が資産 | コードはAIが生成すればよい。何が正しいかの「規範」は人間が Markdown で管理し続けるべきだ。 |
| 立法者としての人間 | 技術的詳細はAIに任せ、人間は「やっていいか」のルールを磨くことに専念する。 |
| デフォルト禁止 | 初期状態は常に実行不可。照合条件を全て満たして初めて実行可能になる。 |
| 説明可能性 | 止まった理由が仕様書に基づいて言語化される。事後監査が可能になる。 |

本実装は「構造化仕様に基づく階層的適合判定および外部確定処理の制御方法」（特許出願済み・日本）を応用したものである。
